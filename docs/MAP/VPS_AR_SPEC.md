> 적용 서비스: MAP 프로젝트 (we-meet-trip)
> 최종 갱신: 2026-06-26 22:00
> 상태: 설계 단계 (미구현)

---

## 개요

여행지에서 AR 앵커(텍스트·포인트)를 장소에 고정시키는 VPS(Visual Positioning System) 기반 기능.
Immersal 방식의 crowd-sourced mapping으로 사전 전체 스캔 부담을 분산.
사용자는 여행 중 1회성으로 가볍게 즐기는 것을 전제로 설계.

---

## 이용 흐름

### 1. 좌표계 선택
- 지도 위에 점으로 기존 좌표계(맵) 목록 표시
- 사용자가 매칭할 좌표계 선택

### 2. 맵 파일 다운로드
- 선택한 좌표계의 맵 파일을 기기에 사전 다운로드
- on-device 방식으로 실시간처럼 동작 (네트워크 지연 제거)

### 3. 부분 스캔 → 로컬 맵 생성 → 서버 맵 매칭
- 사용자가 현재 위치를 부분 스캔
- 로컬 맵 생성 후 다운로드된 서버 맵과 매칭 시도
- **매칭 성공**: 위치 확정, 다음 단계 진행
- **매칭 실패**: 새로운 좌표계 자동 생성 (신규 장소 첫 스캔으로 처리)
- 방식: **Crowd-sourced mapping** — 여러 사용자 스캔을 서버에서 병합해 맵 개선

> **미해결 과제**: 조명 의존성 감소 방법 필요.
> 현재 VPS는 특징점 추출이 조명 변화에 취약. 조명 불변 특징점(예: LiDAR 활용, 야간 보조 조명 힌트) 검토 필요.

### 4. AR 앵커 작성

글씨를 텍스트로 변환하면 글씨체와 주변 맥락이 사라진다. **폴리라인 원본 데이터**를 저장해 실제 손글씨를 그대로 남긴다.

**저장 방식 — 인접 객체 기준 상대 6DOF 폴리라인:**

법선 벡터만 저장하면 렌더링 방향이 Immersal 절대 좌표에 종속된다.  
객체를 기준으로 위치와 방향 모두 상대적으로 저장하면, 나중에 그 객체가 다시 보일 때 VPS 오차와 무관하게 정확히 같은 모양으로 나타난다.

**쓸 때 (Write):**
1. 손가락 궤적 추적 (MediaPipe Hand Landmark, 검지 끝 3D)
2. ARKit/ARCore가 감지한 **인접 객체(평면·특징점 클러스터)의 포즈**를 Immersal 좌표계 기준으로 기록 → `ref` (위치 + 방향)
3. 앵커 좌표계를 그 객체의 로컬 프레임 기준으로 표현 → `rel` (상대 위치 + 상대 방향)
4. 획을 앵커 로컬 프레임의 2D 평면(u, v)으로 기록

**볼 때 (View):**
1. VPS 로컬라이제이션으로 대략 위치 파악
2. `ref.pos`(Immersal 기준)에 해당하는 인근에서 동일 객체 탐지
3. 탐지된 객체 포즈 + `rel` 트랜스폼 → 앵커 월드 포즈 계산
4. 획(u, v)을 앵커 로컬 평면에 투영 → 렌더링

```
저장 데이터 (anchor 1개):
{
  "ref": {
    "pos": [x, y, z],         // 기준 객체 중심 위치 (Immersal 공간)
    "rot": [qx, qy, qz, qw]   // 기준 객체 방향 (Immersal 공간, 단위 쿼터니언)
  },
  "rel": {
    "pos": [dx, dy, dz],      // 객체 로컬 프레임 기준 앵커 위치 오프셋
    "rot": [qx, qy, qz, qw]   // 객체 로컬 프레임 기준 앵커 방향
  },
  "strokes": [                 // 폴리라인 배열
    [[u1,v1], [u2,v2], ...],   // 획 1 (앵커 로컬 평면 내 2D 미터 단위)
    [[u1,v1], [u2,v2], ...]    // 획 2
  ],
  "stroke_width": 0.003        // 선 굵기 (미터)
}
```

**장점:**
- 글씨체·필압·형태 그대로 보존 (텍스트 변환 없음)
- 방향도 객체 기준 상대값 → VPS drift가 글씨 기울기에 영향 없음
- 객체가 탐지되면 절대 좌표 없이도 정확한 위치에 렌더링
- 객체 탐지 실패 시 `ref.pos`(Immersal 절대) + `rel` 조합으로 fallback 가능

### 5. 맵 업데이트 업로드
- 스캔으로 보강된 맵 파일을 서버에 업로드
- 서버에서 기존 맵과 병합 (crowd-sourced 기여)

### 6. 앵커 확인
- 매칭할 좌표계 선택 → 카메라로 해당 부분 비추기
- 위치 매칭 후 앵커 렌더링

## 과거 씬 오버레이 (visitor_log 기능)

앵커를 볼 때 **앵커 작성 당시의 주변 환경**을 AR로 덧씌워 보여준다.  
글씨와 그것이 쓰여진 순간의 공간이 함께 기록·재생되는 "시간 캡슐" 효과.

### 카메라→3D 파이프라인 (앵커 작성 시)

```
ARKit ARFrame / ARCore Frame
├── sceneDepth.depthMap   Float32 per-pixel (미터)
├── capturedImage          YCbCr → RGB
└── camera.intrinsics      [fx, fy, cx, cy]

역투영:
  for pixel (u, v), depth d:
      X = (u - cx) / fx * d
      Y = (v - cy) / fy * d   ← 카메라 공간
      Z = d

× camera_pose (Immersal 로컬라이제이션)
→ 월드 공간 컬러 포인트 클라우드 (x, y, z, r, g, b) × N점
→ 복셀 2cm 다운샘플
→ zstd 압축 → visitor_log 저장
```

### 공간 알파 블렌딩 (렌더링 시)

```
현재 카메라 픽셀에서:
  dist = 최근접 앵커까지 월드 거리

  α = 1 - smoothstep(inner_r=0.3m, outer_r=1.2m, dist)

  two anchors: α = max(α1, α2)     ← bubble 겹침 시 섞임

  out_color = mix(현재_카메라, 과거_씬_투영, α)
```

| 구역 | α | 표시 |
|------|---|------|
| 앵커 0.3m 이내 | 1.0 | 과거 씬 100% |
| 0.3m ~ 1.2m 전환 | SmoothStep | 두 씬 블렌딩 |
| 1.2m 이상 | 0.0 | 현재 카메라만 |
| 두 앵커 bubble 겹침 | max(α1, α2) | 두 과거 씬 조각 공존 |

### 구현 위치

| 컴포넌트 | 레포 | 역할 |
|---------|------|------|
| 씬 캡처 (카메라→포인트클라우드) | `map-service-client` (Flutter/Swift) | ARKit depth 접근, 역투영, 업로드 |
| 씬 저장·서빙 | `visitor_log` (Python FastAPI) | 압축·저장·다운로드 API |
| 공간 블렌딩 렌더링 | `map-service-client` (Metal/GLSL 셰이더) | GPU 알파 블렌딩 |
