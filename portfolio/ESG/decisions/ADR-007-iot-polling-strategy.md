# ADR-007: IoT Adaptive Polling 전략

**상태:** 결정됨 (미구현 — IoT-01~05 태스크 대기 중)
**날짜:** 2026-05-26
**관련:** [ADR-005 IoT 선설계](./ADR-005-iot-design.md)

---

## 배경

Tuya Wi-Fi 16A 스마트 플러그를 통해 세탁기 작동 여부를 감지한다.
Tuya Open API 무료 플랜은 **26,000 API 호출/월**로 제한된다.
17대 세탁기를 대상으로 단순 폴링 시 quota를 크게 초과하므로 최적화가 필요하다.

---

## 하드웨어 로드맵

| 단계 | 하드웨어 | 전환 조건 |
|------|----------|---------|
| 프로토타입 | Tuya Wi-Fi 16A 스마트 플러그 | 초기 소규모 배포 |
| 확장 | Zigbee 스마트 플러그 + Zigbee 허브 | Wi-Fi AP 간섭 또는 관리 대수 30대 초과 시 |

Zigbee 전환 시 앱 서버 인터페이스(REST endpoint + X-Device-Key) 변경 없음.
허브가 Tuya Cloud와 통신하므로 백엔드 코드 수정 불필요.

---

## API Quota 제약

| 항목 | 값 |
|------|---|
| 무료 플랜 한도 | 26,000 호출/월 |
| 일 환산 | 약 866 호출/일 |
| 유료 전환 기준 | 실 사용자 유입 후 월 quota 80% 초과 시 검토 |

## 설계 옵션 비교

### Option 1 — 개별 Adaptive Polling (기각)

각 세탁기별 독립 polling. idle → 빠른 interval, running → 느린 interval.

**문제:** Batch API 미사용 → 17대 × 개별 호출. 기본 17× 비용. Quota 불가.

### Option 2 — 세탁실 Batch + 시간대 감소 (기각)

Batch API로 전체 상태 1회 호출 + 심야 호출 빈도 감소 + 이후 요일별 통계 반영.

**문제:** 세탁기 상태가 아닌 세탁실 단위 동일 interval → Mode B(경쟁 상황)에서도 느린 응답.
UX 최적화 불충분.

### Option 3 — 하이브리드 Adaptive Polling (채택)

Batch API + **가용 세탁기 수(Mode) 기반 interval** + 시간대 감소.

## 최종 설계: 하이브리드 Adaptive Polling

### UX 비대칭성: 점유 감지 vs 완료 감지

세탁기 상태 변화에는 두 종류가 있으며, 감지 지연의 영향이 비대칭적이다:

| 변화 방향 | 늦게 감지했을 때 | 사용자 기대 위반 |
|----------|---------------|--------------|
| 사용 가능 → **사용 중** (점유) | 빈 것으로 표시된 기기에 헛걸음 | **높음** — 이동 비용 이미 발생 |
| 사용 중 → **완료** (멈춤) | 완료된 기기를 조금 늦게 파악 | **낮음** — 이동 전 단계이므로 허용 |

**이것이 설계의 핵심 전제다.**
세탁기가 멈추는 것을 늦게 파악해도 사용자는 아직 이동하지 않은 상태이므로 추가 비용이 없다.
반면 세탁기가 사용되기 시작하는 것을 늦게 파악하면 사용자가 이미 헛걸음을 한 뒤다.

이 비대칭성은 서비스 UX를 넘어 **기숙사 전체의 세탁기 이용 효율**에 직결된다:
점유 감지 지연 → 다수 사용자 헛걸음 → 세탁 포기 또는 재시도 지연 → 세탁기 가동률 저하 및 피크 타임 대기 시간 증가.

따라서 polling 설계의 우선순위는 **"사용 중으로 전환되는 순간을 빠르게 감지하는 것"** 이며,
available 대수가 많은 상태(Mode A)일수록 이 비대칭성이 두드러진다.

### 기본 원리

> **available 대수가 많을수록 polling 빠르게**

이유: available이 많을수록(Mode A) 사용자가 실시간 상태를 보고 이동을 결정한다 — 점유 감지 지연이 직접 헛걸음으로 이어진다(위 UX 비대칭성 참조).
Mode B/C에서는 시스템이 소프트 예약/대기열을 관리하므로 실시간 polling의 UX 기여가 상대적으로 낮다.

| 모드 | available 수 | 낮 interval | 비고 |
|------|-------------|------------|------|
| Mode A | 4대 이상 | 1분 | 헛걸음 방지 최우선 |
| Mode B | 1~3대 | 2분 | 소프트 예약으로 UX 보완 |
| Mode C | 0대 | 8분 | 대기열 알림이 primary notification path |

**심야 (22:00~07:00):** 전 모드 15분으로 감소.

### 층별 공식

단순 mode-level이 아닌 **층별 available 비율**을 반영:

```
polling_interval = base_interval × (1 + α × available_ratio_by_floor)
```

- `available_ratio_by_floor` = 층별 (available / total) 평균
- `α` = 조절 계수 (초기값 1.0, 통계 수집 후 tuning)
- 층별 편차가 클수록 interval 증가 (특정 층만 여유 있을 때 전체 반응성 조절)

### Phase 2 — 통계 기반 조절

초기 운영 데이터 (최소 4주) 수집 후:
- 요일별 × 시간대별 평균 mode 분포 계산
- 예측 interval을 통계 기반으로 사전 조절
- 예: 일요일 저녁은 Mode B 비율 높음 → 해당 시간대 base_interval 단축

## 정량 분석: Quota vs UX 트레이드오프

### 기준 조건

- 세탁기: 17대
- Batch API 사용: **1 API call = 전체 17대 상태** (핵심 가정)
- 무료 quota: 26,000 호출/월

### 나이브 폴링 비교

| 방식 | 계산 | 월 호출 수 |
|------|------|----------|
| 개별 × 1분 (배치 없음) | 17 × 60 × 24 × 30 | 734,400 — **28× 초과** |
| 배치 × 1분 (균일) | 1 × 60 × 24 × 30 | 43,200 — **67% 초과** |
| 배치 × 5분 (균일) | 1 × 12 × 24 × 30 | 8,640 — quota 내, UX 저하 |

### 하이브리드 설계 추정

사용 분포 가정 (경험적 추정, 실 운영 후 보정):

| 시간대 | 시간 | Mode A | Mode B | Mode C |
|--------|------|--------|--------|--------|
| 낮 (07:00~22:00) | 15hr | 40% | 35% | 25% |
| 심야 (22:00~07:00) | 9hr | 10% | 20% | 70% |

**낮 호출 (900분):**
- Mode A (360분, interval 1분): **360 호출**
- Mode B (315분, interval 2분): **158 호출**
- Mode C (225분, interval 8분): **28 호출**
- 소계: **546 호출/일**

**심야 호출 (540분, interval 15분):**
- 540 / 15 = **36 호출/일**

| 항목 | 값 |
|------|---|
| 일 총계 | 582 호출 |
| 월 총계 | **17,460 호출** |
| 무료 quota 대비 | **67%** |
| 잔여 margin | 8,540 호출/월 (33%) |
| Mode A 최대 지연 | 1분 |
| Mode B 최대 지연 | 2분 (소프트 예약으로 보완) |
| Mode C 최대 지연 | 8분 (대기열 알림이 primary) |

### 포트폴리오 기록 요소

이 분석은 포트폴리오에 다음 형태로 정리:

1. **문제 정의:** 단순 폴링 28× quota 초과 → 설계 필요
2. **접근:** Batch API × Mode별 adaptive interval × 시간대 감소
3. **정량 결과:** 734,400 → 17,460 호출/월 (97.6% 절감)
4. **UX 트레이드오프:** Mode A 1분 지연 / Mode B 2분 지연 vs quota 67% 사용
5. **margin 활용:** 잔여 33%는 Phase 2 통계 API 비용 + 트래픽 급증 buffer

## 사용자 / Admin 공개

### 사용자 화면 (DashboardPage 하단)

```
세탁기 정보는 현재 약 N분마다 업데이트됩니다.
```

N = 현재 Mode × 시간대에 따른 실제 interval.

### Admin 패널 (AdminPage.tsx 통합)

```
현재 polling interval : X분
  ├── Mode (A/B/C)           → base N분
  ├── 시간대 (낮/심야)         → ×배율
  └── 층별 available 비율      → +보정
이번 달 API 호출 : NNNN / 26,000
예상 월말 사용량  : NNNN
```
