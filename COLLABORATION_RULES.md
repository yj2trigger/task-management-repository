# Claude 협업 규칙

> 작성일: 2026-05-23
> 적용 범위: task-management-repository를 통해 관리되는 모든 프로젝트

---

## 규칙

### 1. 작업 시작 전 — 모호한 점 질문하기

작업을 시작하기 전, 불명확하거나 해석이 여러 갈래인 부분이 있으면 진행하지 않고 먼저 질문합니다.

- 요청의 범위가 불분명할 때
- 어떤 파일/레포에 반영해야 할지 불확실할 때
- 기존 내용과 충돌 가능성이 있을 때
- 의도한 결과가 여러 가지로 해석될 때

### 2. 작업 전 — 계획 설명 및 이해 일치

질문을 통해 충분히 파악한 뒤, 진행할 작업을 먼저 설명합니다.

- 어떤 파일을 어떻게 수정할 것인지
- 예상되는 결과가 무엇인지
- 주의사항이 있다면 함께 언급

설명 후 사용자의 확인을 받은 뒤 작업을 시작합니다.

### 3. 작업 후 — 결과 정리 및 승인 요청

한 단계의 작업이 끝나면 다음 단계로 넘어가기 전에 반드시 멈춥니다.

- 완료한 작업 내용을 간결하게 정리
- 변경된 파일 목록 및 핵심 변경사항 요약
- 다음 단계 제안 (진행 여부는 사용자가 결정)
- 사용자의 승인을 받은 뒤에만 다음 단계 진행

### 4. 테스트 — 기능 구현과 동시에 작성

기능을 구현할 때 반드시 단위 테스트를 함께 작성합니다.

#### 백엔드 (pytest)

- 각 기능의 service/repository 로직에 대한 단위 테스트를 함께 작성
- 테스트 파일 위치: `backend/tests/test_<기능명>.py`
- 테스트는 실제 DB가 아닌 인메모리 SQLite 또는 테스트 DB 픽스처 사용
- 기능 구현 완료 기준: 코드 + 테스트 통과

**로컬 실행 방법:**
```bash
# project/ 디렉토리에서
docker-compose up db -d        # 테스트용 DB 실행
cd backend
pytest tests/ -v               # 전체 테스트
pytest tests/test_machines.py  # 특정 파일만
```

#### 프론트엔드 (Vitest)

- 프론트엔드는 store(Zustand) 로직과 유틸 함수에 대한 단위 테스트 작성
- UI 컴포넌트 테스트는 복잡도 대비 효과가 낮으므로 프로토타입 단계에서는 생략
- 테스트 파일 위치: `frontend/src/__tests__/`

**로컬 실행 방법:**
```bash
# project/frontend/ 디렉토리에서
npm run test
```

#### GitHub Actions (골격 준비 — 나중에 활성화)

- `ci.yml`은 레포에 미리 작성해 두되, 현재는 수동 실행(`workflow_dispatch`)으로만 트리거
- 팀 협업 또는 실배포 단계 전환 시 `pull_request` 트리거로 변경

### 5. 레포지토리 접근 권한

#### 쓰기 가능 (read + write)

| owner | 레포 |
|-------|------|
| `yj2trigger` | `task-management-repository` |
| `yj2trigger` | `pmg-ic-pbl` |
| `yj2trigger` | `ESG` |
| `yj2trigger` | `storage` |

#### 읽기 전용 (read only) — 절대 쓰기 금지

| owner | 레포 | 비고 |
|-------|------|------|
| `we-meet-trip` | `map-service-client` | 팀 프론트엔드 |
| `we-meet-trip` | `map-service-user` | 팀 사용자/인증 |
| `we-meet-trip` | `map-service-hub` | 팀 API Gateway |
| `we-meet-trip` | `map-service-infra` | 팀 인프라 |
| `we-meet-trip` | `map-service-agent` | 팀 AI 에이전트 |

> `we-meet-trip` organization의 모든 레포는 참고/읽기만 가능합니다.
> 어떤 상황에서도 이 레포들에 커밋, 파일 생성, 파일 수정을 하지 않습니다.

### 6. 분석 및 리뷰 — 수집한 데이터 기반으로 결론 직접 기술

코드 리뷰, 문서 검토, 상태 파악 등 분석 작업 시 다음 원칙을 따릅니다.

- **이미 수집한 데이터로 확인 가능한 사항은 결론을 직접 기술한다.**
  "확인을 권장합니다", "직접 확인이 필요합니다" 같은 표현은 실제로 확인이 불가능할 때만 사용합니다.
  수집한 diff, 커밋, 코멘트 등으로 판단할 수 있는 사항은 즉시 결론을 내립니다.

- **"다음 단계에서 확인할 사항"에는 현재 수집한 데이터로 판단 불가능한 것만 기재한다.**
  이미 확인된 사항이나 본문에서 이미 언급한 내용을 중복으로 기재하지 않습니다.

## 흐름 요약

```
[모호한 점 질문]
      ↓
[작업 계획 설명 + 테스트 계획 포함]
      ↓
[승인]
      ↓
[기능 구현 + 테스트 코드 동시 작성]
      ↓
[로컬 테스트 통과 확인]
      ↓
[결과 정리 (구현 내용 + 테스트 결과 요약)]
      ↓
[승인] → [다음 단계]
```

## 예외

긴급하거나 명백히 단순한 작업(예: 오타 수정, 단순 상태 업데이트)은 계획 설명을 간략히 할 수 있습니다.
단, 결과 정리 및 승인 요청은 항상 지킵니다.
