# Claude 협업 규칙

> 작성일: 2026-05-23
> 적용 범위: task-management-repository를 통해 관리되는 모든 프로젝트

---

## 규칙

### 1. 작업 시작 전 — 모호한 점 질문하기

작업을 시작하기 전, 불명확하거나 해석이 여러 갈래인 부분이 있으면 진행하지 않고 먼저 질문합니다.

- 요청의 범위가 불분명할 때
- 어떤 파일/레포에 반영해야 할지 불확실할 때
- 기존 내용과 충돌 가능성이 있을 때
- 의도한 결과가 여러 가지로 해석될 때

### 2. 작업 전 — 계획 설명 및 이해 일치

질문을 통해 충분히 파악한 뒤, 진행할 작업을 먼저 설명합니다.

- 어떤 파일을 어떻게 수정할 것인지
- 예상되는 결과가 무엇인지
- 주의사항이 있다면 함께 언급

설명 후 사용자의 확인을 받은 뒤 작업을 시작합니다.

### 3. 작업 후 — 결과 정리 및 승인 요청

한 단계의 작업이 끝나면 다음 단계로 넘어가기 전에 반드시 멈춥니다.

- 완료한 작업 내용을 간결하게 정리
- 변경된 파일 목록 및 핵심 변경사항 요약
- 다음 단계 제안 (진행 여부는 사용자가 결정)
- 사용자의 승인을 받은 뒤에만 다음 단계 진행

### 4. 테스트 — 기능 구현과 동시에 작성

기능을 구현할 때 반드시 단위 테스트를 함께 작성합니다.

#### 백엔드 (pytest)

- 각 기능의 service/repository 로직에 대한 단위 테스트를 함께 작성
- 테스트 파일 위치: `backend/tests/test_<기능명>.py`
- 테스트는 실제 DB가 아닌 인메모리 SQLite 또는 테스트 DB 픽스처 사용
- 기능 구현 완료 기준: 코드 + 테스트 통과

**로컬 실행 방법:**
```bash
# project/ 디렉토리에서
docker-compose up db -d        # 테스트용 DB 실행
cd backend
pytest tests/ -v               # 전체 테스트
pytest tests/test_machines.py  # 특정 파일만
```

#### 프론트엔드 (Vitest)

- 프론트엔드는 store(Zustand) 로직과 유틸 함수에 대한 단위 테스트 작성
- UI 컴포넌트 테스트는 복잡도 대비 효과가 낮으므로 프로토타입 단계에서는 생략
- 테스트 파일 위치: `frontend/src/__tests__/`

**로컬 실행 방법:**
```bash
# project/frontend/ 디렉토리에서
npm run test
```

#### GitHub Actions (골격 준비 — 나중에 활성화)

- `ci.yml`은 레포에 미리 작성해 두되, 현재는 수동 실행(`workflow_dispatch`)으로만 트리거
- 팀 협업 또는 실배포 단계 전환 시 `pull_request` 트리거로 변경

### 5. 브랜치 & 이슈 워크플로우 (ESG 레포)

#### 브랜치 전략

```
main          → 프로덕션 (owner는 직접 push 가능, 팀원은 PR 필수)
feature/xxx   → 기능 개발
issue/xxx     → 버그 수정
```

#### feature 브랜치

```bash
git checkout -b feature/기능명    # main에서 분기
# 작업 후
git push origin feature/기능명
# GitHub에서 PR 생성 → main으로
```

#### 이슈 & 버그 수정

1. GitHub Issues에 버그/개선 사항 등록
2. `issue/이슈번호-간단설명` 브랜치 생성
3. 수정 후 PR → main
4. PR 설명에 `Closes #이슈번호` 기재 (merge 시 이슈 자동 닫힘)

```bash
git checkout -b issue/42-fix-queue-notified
# 수정 후
git push origin issue/42-fix-queue-notified
# PR 본문에: Closes #42
```

#### PR 조건 (팀원)

- CI 통과 필수 (Backend Tests + Frontend Tests)
- 1명 승인 필수
- owner는 bypass 가능

### 7. MAP 프로젝트 문서 — 원격 저장소 반영 명시적 승인 필수

**이 레포(task-management-repository)에 포함된 MAP 프로젝트 관련 문서는 사용자의 명시적 승인 없이 원격 저장소에 반영하지 않습니다.**

#### 적용 대상

MAP 프로젝트와 관련된 모든 파일 및 디렉토리:

- `docs/` 하위의 MAP 관련 문서
- `reports/` 하위의 MAP 관련 보고서
- `tasks/` 하위의 MAP 관련 태스크
- `portfolio/` 하위의 MAP 관련 자료
- MAP 프로젝트를 언급하거나 참조하는 모든 파일

#### 금지 행위 (명시적 승인 없이 절대 불가)

| 행위 | 설명 |
|------|------|
| `git push` | 로컬 커밋을 원격에 반영 |
| PR 생성 | GitHub Pull Request 생성 |
| 파일 직접 업로드 | GitHub API/UI를 통한 파일 생성·수정 |
| 브랜치 push | 어떤 브랜치로든 push |

#### 허용 행위

- 로컬에서 파일 작성, 수정, 커밋 (`git commit`) — 원격 반영 없음
- 파일 내용 검토, 분석, 읽기
- 사용자가 명시적으로 "push해", "PR 올려", "업로드해" 등을 지시한 경우에만 원격 반영

#### 승인 확인 방법

작업 중 MAP 프로젝트 문서를 원격에 반영해야 할 상황이 생기면:

1. 반드시 먼저 멈춘다
2. 사용자에게 "MAP 프로젝트 문서 [파일명]을 원격에 반영해도 될까요?" 라고 묻는다
3. 명시적 허가를 받은 뒤에만 진행한다

> **이유:** MAP 프로젝트는 팀 협업 레포(`we-meet-trip`)에서 관리되며, 관련 문서의 외부 노출 및 비승인 반영은 팀 워크플로우와 충돌할 수 있습니다.

### 8. 분석 및 리뷰 — 수집한 데이터 기반으로 결론 직접 기술

코드 리뷰, 문서 검토, 상태 파악 등 분석 작업 시 다음 원칙을 따릅니다.

- **이미 수집한 데이터로 확인 가능한 사항은 결론을 직접 기술한다.**
  "확인을 권장합니다", "직접 확인이 필요합니다" 같은 표현은 실제로 확인이 불가능할 때만 사용합니다.
  수집한 diff, 커밋, 코멘트 등으로 판단할 수 있는 사항은 즉시 결론을 내립니다.

- **"다음 단계에서 확인할 사항"에는 현재 수집한 데이터로 판단 불가능한 것만 기재한다.**
  이미 확인된 사항이나 본문에서 이미 언급한 내용을 중복으로 기재하지 않습니다.

